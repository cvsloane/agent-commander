How it works

Agent Commander keeps `agentd` and sessions as the execution substrate, then layers memory and orchestration on top.

1. agentd discovers tmux

agentd runs on each host, polls tmux panes, captures snapshots, and streams output.

2. Control plane adds memory + orchestration

REST + WebSocket APIs persist sessions, approvals, automation runs, work queues, and scoped memory in Postgres.

3. Dashboard stays session-native

The UI gives you tmux, sessions, automation, memory, and approvals without inventing a second runtime.

4. Optional outer systems integrate cleanly

Hermes, OpenAI summaries, Deepgram transcription, and OpenClaw notifications plug in as needed.

FAQ

Is there a public demo?

No. The public site is docs only. The live app runs on private infrastructure.

What inspired the new architecture?

Paperclip informed the orchestration model and Ruflo informed the memory model, but Agent Commander still runs on its own tmux-and-session-native stack.

What does it require?

tmux, Postgres, Node 20+, and Go 1.22+. See the getting started guide.

Can I run it on one machine?

Yes. The control plane and dashboard can run together with local Postgres.
